## Session Insights

### Problems Found and Their Root Causes

**1. `FAVICON_SVG` undefined constant**
- _Root cause_: The format string in `render_html` referenced `FAVICON_SVG` but the constant was
  never declared. It was intended from design notes but omitted during initial implementation.
- _Fix_: Added `const FAVICON_SVG: &str = concat!(...)` before `ARCHITECTURE_DIAGRAM`.
- _Lesson_: When a large format string references Rust consts, check that every `{CONST_NAME}`
  interpolation has a matching `const` definition — easy to miss if the const was planned but
  not yet written.

**2. Rust raw string `r#"..."#` terminated by `href="#"`**
- _Root cause_: Navbar anchor tags contain `href="#">`, which includes the literal sequence `"#`.
  In a `r#"..."#` raw string literal, `"#` is the terminator, so Rust ends the string
  mid-HTML and the remainder is invalid syntax.
- _Fix_: Changed the entire `render_html` format string delimiter from `r#"..."#` to
  `r##"..."##`. The double-hash delimiter only terminates at `"##`, so single `"#` sequences
  inside are safe.
- _Lesson_: Any Rust raw string that embeds HTML with anchor fragment URLs (`href="#..."`)
  or URLs containing `#` must use at least `r##"..."##`. The compiler error message points to
  a random-looking location after the anchor tag, making the root cause non-obvious.

**3. Test fixtures not updated after struct field additions**
- _Root cause_: `Snapshot` and `InventoryFile` gained new fields in the previous session. Rust
  struct literal syntax (without `..Default::default()`) requires all fields to be listed, so
  every `Snapshot { ... }` and `InventoryFile { ... }` literal in tests became a compile error.
- _Fix_: Added the missing fields (`config_files: vec![]`, `content_preview: None/Some(...)`)
  to all affected test fixtures. Added `Default` impl to `Snapshot` so future additions can use
  `..Default::default()` in fixtures.
- _Lesson_: When adding non-optional fields to structs, proactively grep for all struct literal
  instantiation sites (not just usages) to find fixtures that need updating before the first
  compile attempt.

**4. `api.rs` test helper missing new struct fields**
- _Root cause_: `ClaudeClient` struct had `use_subprocess` and `claude_path` added (for the
  Claude CLI subprocess feature) but the `test_client()` test helper used a struct literal and
  wasn't updated. This is the same "test fixture drift" pattern as #3.
- _Fix_: Added `use_subprocess: false, claude_path: String::new()` to the helper.
- _Lesson_: Struct literal test helpers are fragile when the struct is actively evolving. Adding
  `#[derive(Default)]` or a builder pattern makes these helpers forward-compatible.

**5. DOM mutation destroys event listener on close button**
- _Root cause_: The architecture detail panel JS used `panel.innerHTML = newContent` to update
  what's displayed when a node is clicked. `innerHTML` completely replaces all child DOM nodes,
  including the close button. After the first node click, the close button's element was
  destroyed and a new one created without the attached event listener — clicks on it did nothing.
- _Fix_: Changed the JS to target an inner `<div id="arch-detail-content">` for `innerHTML`
  updates, leaving the outer `<div id="arch-detail">` (and its close button child) untouched.
- _Lesson_: Never `innerHTML`-replace a container that holds interactive children. Always target
  a dedicated content div, or use `replaceChildren()` / `textContent` on leaf nodes.

**6. JS string escaping for embedded file content**
- _Root cause_: File content is embedded at HTML generation time as JS string literals:
  `registerFileContent('key', 'content here')`. Rust strings may contain backslashes, single
  quotes, and newlines — all of which have special meaning inside a JS single-quoted string.
- _Fix_: Added `js_string_escape(s: &str) -> String` helper that escapes `\\`, `\'`, `\n`, `\r`
  before embedding any file content.
- _Lesson_: Any time you interpolate arbitrary text into a JS string literal at codegen time
  (in any language), write a dedicated escape function first — don't discover missing escapes
  from a broken preview.

**7. Test assertions not updated after event label format change**
- _Root cause_: The new `event_label_badge` function strips the `tool_use(...)` wrapper and
  renders just the inner tool name as a colored badge. Two tests expected the old
  concatenated format (`"tool_use(Read)"`, `"tool_use(&lt;Read&amp;Write&gt;)"`).
- _Fix_: Updated assertions to check for the actual badge output: CSS class `ev-tool`, badge
  text `>Read<`, escaped inner name `&lt;Read&amp;Write&gt;`.
- _Lesson_: When changing the rendering format of a component, grep for all test assertions
  that reference old format strings before running tests — otherwise each failed test is a
  surprise rather than an expected cleanup.
