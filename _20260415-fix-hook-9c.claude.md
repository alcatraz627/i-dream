## Session Insights

### Problems Encountered and Root Causes

**Problem 1 — `cargo` not in PATH**

- **Symptom:** `cargo install --path .` → "command not found"
- **Root cause:** Claude Code's session `$PATH` is stripped down and does not include
  `~/.cargo/bin`, even on machines where the interactive shell has it
- **Fix:** Always use full path `~/.cargo/bin/cargo` for Rust toolchain commands in this
  context
- **Pattern:** This will recur on every session. Keep `~/.cargo/bin/` prefix as muscle
  memory for Rust commands.

**Problem 2 — `socat` not installed; all hook events silently lost**

- **Symptom:** `hooks install` succeeded; hook files were present; but no events reached
  the daemon and no errors appeared in logs (hooks suppress stderr with `2>/dev/null`)
- **Root cause:** `socat` is a third-party Unix utility commonly used for socket bridging
  but not bundled with macOS. The hook scripts used it to pipe shell output to a Unix
  domain socket, assuming it was installed.
- **Why silent:** All four hook scripts had `|| true` after the socat pipe, and stderr was
  suppressed. A missing binary causes "command not found" on stderr, which was discarded.
- **Fix:** Replaced `socat` with Python 3's built-in `socket` module
  (`socket.AF_UNIX`). Python 3 is always present on modern macOS; `socket.AF_UNIX` is
  in stdlib.
- **For `user_prompt_submit` specifically:** The Python code sends directly from inside the
  heredoc using `IDREAM_SOCKET="$SOCKET"` env var (can't reference `$SOCKET` bash variable
  inside a quoted heredoc `<< 'PYEOF'`). This also removes the two-step
  stdout→pipe→socat chain and is cleaner.
- **Broader lesson:** Any hook script that uses `|| true` with a non-standard tool will
  fail silently. Always test with the exact hook invocation, not just the logic.

**Problem 3 — Old daemon binary running after `cargo install`**

- **Symptom:** `user_signal` payload sent successfully; no `signals.jsonl` created; daemon
  log showed deserialization error for `user_signal` variant
- **Root cause:** `cargo install` replaces the binary _file_ on disk, but the already-running
  daemon process (`i-dream start`) was exec'd from the old binary and continues running it
  in memory. The new `HookEvent::UserSignal` enum variant only exists in the new binary.
- **Fix:** `i-dream stop` + `i-dream start` to relaunch with the updated binary.
- **Key insight:** This is standard Unix behavior — replacing an executable on disk does
  not affect running processes. Always restart long-running daemons after `cargo install`.

**Problem 4 — `user-prompt-submit.sh` missing after initial `hooks install`** (prior session)

- **Symptom:** `hooks install` reported success; file not in hooks directory
- **Root cause:** `~/.cargo/bin/i-dream` was the previously installed binary, which
  predated the `write_user_prompt_submit_hook()` function added in this feature branch.
  `cargo install --path .` had not been run yet to update the installed binary.
- **Fix:** Always run `cargo install --path .` before `i-dream hooks install` when the
  installed binary is out of date with the source tree.

**Non-issue — `os error 57` in daemon log**

- The daemon writes a response back on the socket after processing each event. For
  fire-and-forget hook clients (post_tool_use, stop, user_prompt_submit), the client
  closes the socket immediately after `sendall`, before the daemon can respond. This
  produces `WARN Hook event handler failed: Socket is not connected (os error 57)`.
- This is **harmless and expected**. The signal was already written to `signals.jsonl`
  before the response write attempt.
