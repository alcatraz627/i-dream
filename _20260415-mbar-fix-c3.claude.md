## Session Insights

### Problems Found and Their Root Causes

**Problem 1 — `phase_skipped` in dashboard**

- **Symptom:** Dashboard shows `phase_skipped: no new sessions to consolidate` for every item
- **Actual cause:** Two different mechanisms emit `PhaseSkipped`:
  1. `dreaming.rs:239-250` — SWS emits when `summaries.is_empty()` (genuine idle: no sessions)
  2. `dreaming.rs:785-819` — SWS/REM/Wake emit when API call fails ("budget exhausted")
- The dashboard shows last 5 cycles (constant `RECENT_TRACES_LIMIT = 5` at `dashboard.rs:62`). All 5 were
  API-failure cycles. The "no new sessions" message was from earlier skip-only cycles not visible in the window.
- **Fix:** No code change needed. Top up API credits.
- **Gotcha:** The two PhaseSkipped messages look identical in the dashboard. A future improvement would be to carry
  a `reason` field on the event to distinguish idle-skip from error-skip.

**Problem 2 — Widget Start Daemon button does nothing**

- **Symptom:** Clicking Start Daemon logs `direct start launched PID=XXXXX` but `isDaemonRunning` immediately
  returns false; no pid file ever appears.
- **Root cause:** `main.rs:52-53` — `i-dream start` (no flag) calls `daemon.run_foreground()` which runs the event
  loop but never writes a pid file. The pid file is only written by `daemon.daemonize()`, which is only called when
  the `--daemonize` flag is present (`main.rs:50-51`). The v2 widget launched `["start"]` without the flag.
- **Fix:** Change fallback to `["start", "--daemonize"]` in `startDaemon()`.
- **Why the design is this way:** `daemonize()` is not actually forking — it just adds pid file lifecycle management
  (write on entry, clean up on exit). The naming is confusing but intentional: backgrounding is delegated to the
  supervisor (launchd/widget). `--daemonize` simply enables pid file bookkeeping.
- **Secondary issue:** 4 zombie `i-dream` processes from prior button clicks — `run_foreground()` never exits
  cleanly since nothing called `i-dream stop` and the widget never knew they were running.

**Problem 3 — Two widget instances after `build.sh` when LaunchAgent is installed**

- **Symptom:** Two `i-dream-bar` PIDs after every build. Log shows two `launched PID=XXXXX` entries within seconds.
- **Root cause:** `KeepAlive → SuccessfulExit: false` in the plist means launchd restarts the widget when it is
  SIGTERM'd (signal = non-zero exit). The build script's `pkill -x i-dream-bar` kills the launchd-managed process;
  launchd immediately respawns it with the OLD binary (still compiling). Then `open "$OUTPUT"` launches a second
  instance once the new binary is ready.
- **Fix in build script:** At script start, check if `launchctl list "$LABEL"` succeeds. If so:
  1. `launchctl bootout gui/$UID/$LABEL` — removes launchd's awareness of the label, preventing auto-restart
  2. Kill remaining processes normally
  3. Compile + sign
  4. `launchctl bootstrap gui/$UID "$PLIST"` — re-register; launchd starts fresh single instance
     If LaunchAgent is not registered, plain `open "$OUTPUT"` path is used unchanged.
- **Key insight:** `SuccessfulExit: false` is the right KeepAlive policy for crash-restart-only behavior, but it
  means any `pkill` triggers a restart. The solution is always to manage the launchd lifecycle explicitly
  (bootout/bootstrap) rather than killing the managed process directly.

**Problem 4 — `ISO8601DateFormatter` fails on fractional-second timestamps**

- **Symptom:** `fmtDate("2026-04-14T22:19:53.467875Z")` returned "–" (nil date).
- **Root cause:** `ISO8601DateFormatter()` with default options does not parse fractional seconds. The
  `state.json` `last_consolidation` field uses microsecond precision (6 decimal places).
- **Fix:** Added `isoDate()` helper that tries `.withFractionalSeconds` first, falls back to standard ISO8601.
